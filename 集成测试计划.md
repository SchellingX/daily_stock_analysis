# Headless Gemini CLI 集成测试计划 (v0.2.0)

本计划旨在验证 `daily_stock_analysis` 项目中通过子进程（Subprocess）以无头（Headless）模式调用 Gemini CLI 的可行性与稳定性。

## 1. 测试环境准备
- **环境变量**：确保已配置 `GEMINI_API_KEY` 及必要的 `https_proxy`（如适用）。
- **工具版本**：验证 `gemini --version` 确保 CLI 工具已安装且在 `$PATH` 中。
- **参考标准**：遵循 `1-Tutorials/agent/gemini-cli-headless.md`。

## 2. 核心测试用例

### 2.1 基础连通性与参数校验
- **目标**：验证 `-p` 参数能正确进入非交互模式。
- **操作**：执行 `gemini -p "ping"`。
- **预期结果**：Stdout 仅输出文本回复（如 "pong" 或类似），Stderr 无动画或进度条输出。

### 2.2 Stdin 数据流传递 (大规模上下文)
- **目标**：通过 Stdin 将复杂的股票上下文（技术面+新闻）喂给 Gemini。
- **操作**：
  ```python
  context_text = "Stock: AAPL, MA5: 180, News: [Positive report...]"
  res = subprocess.run(["gemini", "-p", "根据前文总结风险"], input=context_text, ...)
  ```
- **预期结果**：AI 能够解析 Stdin 中的长文本并给出精准回复，证明管道传输比命令行参数更稳健。

### 2.3 结构化数据 (JSON) 返回
- **目标**：验证 `GeminiAnalyzer` 所需的 JSON 解析稳定性。
- **操作**：
  ```bash
  gemini -p "返回 AAPL 行业 JSON" --output-format json
  ```
- **预期结果**：输出纯净 JSON，不含 Markdown 反引号。

### 2.4 Headless 鲁棒性与异常处理 (模拟)
- **目标**：测试非交互环境下的错误退出机制。
- **测试点**：
  - **429 (Rate Limit)**：脚本应能捕获 `res.stderr` 中的 429 并配合 Python 层进行指数退避。
  - **Timeout**：验证 `subprocess.run(timeout=120)` 能在 LLM 响应超时时正确释放系统资源。

## 3. 架构集成验证 (New)

### 3.1 协议拦截验证
- **目标**：验证配置 `LITELLM_MODEL=gemini-cli/gemini-2.0-flash` 时，系统能正确路由到 CLI 适配器。
- **测试点**：
  - 模拟 `src/analyzer.py` 调用，拦截 `litellm.completion` 抛出的错误，转向 `_call_gemini_cli_headless`。

### 3.2 环境变量继承 (借鉴 Podcast2md)
- **目标**：验证子进程能正确继承代理配置。
- **测试点**：检查子进程中的 `os.environ` 是否包含 `https_proxy`，确保在复杂网络环境下的连通性。
