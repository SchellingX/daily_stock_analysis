# Daily Stock Analysis - 分阶段测试计划 (v4.0 架构升级)

本测试计划旨在分步骤、安全地验证“多专家共识架构”（Multi-Expert Ensemble Architecture）的集成情况。每完成一个阶段，必须进行代码提交和测试日志（test-log.md）的更新。

---

## 阶段一：Gemini CLI Headless 基础集成 (✅ 已完成)
- **目标**：验证在项目中通过子进程无头调用 `gemini-cli` 的可行性。
- **验收标准**：
  - [x] `-p` 参数能进入非交互模式。
  - [x] Stdin 管道数据能成功注入大段上下文。
  - [x] 成功解析非标准 JSON 输出（清理反引号）。

---

## 阶段二：单一“大师”节点验证 (⏳ 待进行)
- **目标**：验证从 `AI-Hedge-Fund` 提取的大师 Prompt 能在当前项目中正常生成结构化信号。
- **任务**：
  1. 创建 `src/agents/experts/warren_buffett.py`，配置 system prompt。
  2. 编写 `tests/test_single_expert.py`，模拟一段包含基本面和新闻的上下文输入。
- **验收标准**：
  - 大师能根据输入输出标准的 JSON 格式：`{"signal": "...", "confidence": XX, "reasoning": "..."}`。
  - 测试用例必须通过断言验证字段的存在与类型。

---

## 阶段三：并发调用与共识引擎验证 (⏳ 待进行)
- **目标**：验证并发启动多个 `gemini-cli` 实例的稳定性，以及共识引擎（Consensus Aggregator）的聚合逻辑。
- **任务**：
  1. 在 `analyzer.py` 中实现 `asyncio.gather` 并行执行 3-4 个大师实例。
  2. 实现 `ConsensusAgent`，将各大师的输出喂入，输出最终的融合建议。
  3. 编写 `tests/test_consensus_engine.py`。
- **验收标准**：
  - 4 个 `gemini-cli` 子进程能同时运行且互不阻塞。
  - 在极端冲突场景（例如：技术看涨，基本面看跌）下，共识引擎能正常输出权衡后的解释或触发 `Hold`。
  - 并发调用的总耗时不得超过单次调用的 2 倍（主要受限于最慢的节点响应时间）。

---

## 阶段四：战绩追踪与动态权重验证 (⏳ 待进行)
- **目标**：验证 `analyst_track_record` 数据库存储及基于胜率的置信度调整公式。
- **任务**：
  1. 创建轻量级 SQLite 数据库或 JSON 持久化方案。
  2. 编写打分脚本：对比 `(预测日 + 5 天)` 的实际股价变化与 `预测 signal`。
  3. 编写 `tests/test_weighting_engine.py`。
- **验收标准**：
  - 高胜率大师（如历史准确率 80%）的 `Final_Confidence` 显著高于低胜率大师（如 40%）。
  - 当某个大师表现极差时，即使其给出的原始置信度为 `100`，加权后的影响也能被有效削弱。

---

## 阶段五：全量数据接入与 WebUI 渲染 (⏳ 待进行)

  - 共识报告呈现合理。
